# Рекомендации по улучшению системы сборки

## 1. Верификация SDK после сборки

Сейчас после сборки нет проверки целостности SDK. Пропущенные файлы (DLL, `__init__.py`, зависимости) обнаруживаются только при запуске.

Предлагается добавить скрипт верификации, который после сборки проверяет:
- У каждого `.pyd`/`.so` файла есть `__init__.py` в той же директории (или в родительском пакете)
- Все DLL-зависимости каждого `.pyd` разрешимы из `sdk/bin/` и системных путей
- Все модули из `modules.conf` представлены в `lib/cmake/`

Это быстрая проверка, которая сэкономит часы отладки.

## 2. Общий cmake-хелпер для всех библиотек

`TerminModule.cmake` автоматизирует boilerplate для компонентных библиотек, но основные библиотеки (base, scene, render, ...) дублируют одни и те же блоки: MSVC warnings, install rules, GNUInstallDirs, CMakePackageConfigHelpers, RPATH.

Можно создать аналогичный хелпер для основных библиотек — либо расширить `TerminModule.cmake`, либо создать общую функцию `termin_configure_target()`, которая применяет стандартные настройки к уже созданному target.

## 3. Объединение стадий C++ и Python

Сейчас сборка проходит в две стадии: сначала C++ (`TERMIN_BUILD_PYTHON=OFF`), потом Python (`TERMIN_BUILD_PYTHON=ON`). Поскольку включение Python не влияет на C++ targets, можно собирать всё сразу с `TERMIN_BUILD_PYTHON=ON`.

Две стадии оправданы только тем, что Python/nanobind может быть не установлен. Если это не актуально (сборочная среда всегда полная), стадии можно объединить — это сократит время конфигурации cmake.

## 4. Отдельный export-макрос для каждой C-библиотеки

Сейчас все библиотеки используют общий `TC_API` / `TC_EXPORTS`. При сборке, например, termin-render с `TC_EXPORTS`, функции из заголовков termin-base тоже помечаются как `dllexport`. Линковка работает корректно (символы берутся из import lib), но семантически библиотека обещает экспортировать чужие символы.

Правильный подход — отдельный макрос для каждой библиотеки (`TERMIN_BASE_API`, `TERMIN_SCENE_API`, ...), как уже сделано для C++ классов. Это потребует переименования `TC_API` во всех заголовках, но устранит неоднозначность.

Низкий приоритет — текущая схема работает и не вызывает реальных проблем.

## 5. CMake superbuild

Сейчас оркестрация сборки — это bash/powershell скрипты, которые вызывают cmake для каждого модуля в порядке, заданном `modules.conf`. Это работает, но хрупко: порядок задаётся вручную, нет проверки зависимостей, параллелизация невозможна.

CMake `ExternalProject_Add` (superbuild) может взять эту оркестрацию на себя: каждый модуль становится ExternalProject с явными зависимостями, cmake сам определяет порядок и может параллелить независимые модули.

Это существенная переработка, но она убирает целый класс ошибок (неправильный порядок, забытый модуль, неконсистентные флаги между модулями).

## 6. Единый build через add_subdirectory

Самый радикальный вариант: один корневой `CMakeLists.txt`, который подключает все модули через `add_subdirectory()`. Один configure, один build graph, cmake сам разруливает все зависимости.

Преимущества:
- Один `cmake --build` собирает всё
- Полная параллелизация
- Невозможно ошибиться в порядке
- Изменение в заголовке автоматически пересобирает зависимые модули

Главный недостаток: модули перестают быть полностью независимыми CMake-проектами. Их нельзя будет собрать по отдельности без корневого `CMakeLists.txt`. Также усложняется CI, если нужно собирать только часть проекта.

Для текущего масштаба (~30 модулей) superbuild (п.5) — разумный компромисс между независимостью модулей и автоматизацией.
