# Taskboard Guidelines

Короткая памятка для ведения Kanboard-доски Termin.

## Оформление карточек

- Заголовок должен начинаться с области: `[render]`, `[build]`, `[assets]`, `[editor]`, `[qa]`, `[android/openxr]` и т.п.
- Если тип неочевиден, добавляй его в заголовок или тегами: `bug`, `umbrella`, `plan`, `cleanup`, `gate`, `qa`, `manual`.
- Одна обычная карточка должна описывать одну закрываемую работу. Большие направления оформляй как `umbrella` и веди конкретные подзадачи отдельными карточками.
- В описании обязательно должна быть точка закрытия: что должно пройти, исчезнуть, собраться, провериться или быть удалено.
- Не оставляй карточку с формулировкой только "разобраться". Для investigation укажи expected outcome: закрыть как неактуальное, создать bug/fix-card или превратить в конкретную задачу.

## Обновление статуса

- Карточки не должны оставаться открытыми "на всякий случай". Если acceptance выполнен и проверка прошла - закрывай.
- Если по ходу работы остался новый хвост, не держи старую карточку живой: закрой сделанное и создай/оставь отдельную карточку на новый scope.
- Когда комментарии уже меняют смысл карточки, актуализируй description: `Current state`, `Remaining`, `Blocked by`, `Verification`.
- Комментарий используй как журнал события. Description должен оставаться источником актуального состояния.
- `On Test` должен означать конкретное ожидание: Windows smoke, Quest device run, manual editor click/drag, CI job. После проверки закрывай или возвращай в работу с новым дефектом.

## Гигиена доски

- Не плодить дубликаты: перед созданием карточки проверь близкие umbrella/bug/gate.
- Связанные crash/leak/shutdown проблемы лучше группировать под umbrella, но не смешивать разные reproductions в одну fix-card.
- Зонтичные карточки периодически чистить: убрать уже сделанное, перечислить живые children, обновить критерий завершения.
- Если задача стала неактуальной из-за миграции или нового решения, закрыть её с коротким комментарием, а не хранить как исторический артефакт.
- Для ручных и platform-specific проверок явно указывать требуемую среду и команду/сценарий.
