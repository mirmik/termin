Termin build system

Целевая архитектура project build и runtime packaging

Сборка проекта должна стать compiler pipeline: явные входы, фазы, артефакты, проверки и воспроизводимый runtime package для desktop, Android и Quest/OpenXR.

Дата: 2026-06-17 Scope: project build Targets: desktop / android / quest_openxr Связанные задачи: #27-#33, #39-#41

Главная линия

  • Editor запускает сборку и показывает диагностику, но не владеет графом ресурсов.
  • Runtime package является стабильным контрактом между build side и runtime side.
  • Desktop, Android и OpenXR различаются wrapper-ом, а не смыслом project package.
  • Missing resources и несовместимые capabilities должны падать на build stage.

North star

Сборка как compiler pipeline

Сейчас система "как-то собирает" за счет нескольких путей и неявных предположений. Целевая форма должна объяснять каждый выходной файл через профиль, project source tree и capability manifest установленного SDK.

0
InputsProject source, build profile, SDK capabilities, environment.
1
Analyze graphДетерминированный граф сцен, ресурсов, модулей и зависимостей.
2
Compile artifactsScenes, materials, shaders, meshes, Python modules, sidecars.
3
Write packageTarget-independent runtime package с manifest.json.
4
Target wrapperDesktop bundle, Android APK или Quest/OpenXR APK.
5
ValidateSchema, paths, shaders, resources, platform requirements, smoke contract.

Цели

  • Единая модель build profiles для всех targets.
  • Runtime package как версионированный контракт.
  • Build graph из файлов проекта и metadata, не из editor state.
  • Общий package contract для desktop, Android и OpenXR.
  • Ошибки ресурсов, capabilities и SDK обнаруживаются до runtime.

Не-цели

  • Не переписывать весь CMake SDK graph.
  • Не требовать один runtime host для desktop window, Android surface и OpenXR swapchain.
  • Не обещать Python runtime на Android без отдельного решения.
  • Не сохранять бесконечную compatibility с недособранными legacy artifacts.

Boundaries

Два уровня сборки и границы ownership

Важно не смешивать сборку SDK и сборку пользовательского проекта. Этот план про project build, target packaging и runtime package contract.

Contract

Build Profile Contract

Профиль должен быть явным и версионированным. Target dispatch живет в Python backend, а C++ termin_builder только находит проект, грузит профиль, вызывает backend и возвращает exit code.

Путь и targets

project_settings/build_profiles.json

desktop android quest_openxr future: server / web / headless_test

Configurations

  • dev: быстрый local build, возможны явные smoke fallbacks.
  • debug: debuggable artifact, без fake resources.
  • release: strict diagnostics, deterministic resource set.
{
  "version": 1,
  "profiles": {
    "dev": {
      "target": "desktop",
      "configuration": "dev",
      "entry_scene": "Scenes/Main.scene",
      "output_dir": "dist/dev",
      "resource_policy": "dev",
      "python": { "enabled": true },
      "runtime": { "backend": "vulkan" }
    },
    "quest": {
      "target": "quest_openxr",
      "configuration": "debug",
      "entry_scene": "Scenes/XR.scene",
      "output_dir": "dist/quest",
      "resource_policy": "strict",
      "android": { "abi": "arm64-v8a", "platform": "android-26" },
      "openxr": { "required": true }
    }
  }
}

Phases

Фазы project build

Каждая фаза имеет вход, ответственность, выход и диагностические правила. Это делает pipeline тестируемым без editor runtime.

Фаза Ответственность Выход
0. Preflight Проверка schema, entry scene, output dir, SDK capability manifest, shaderc, Gradle/Android SDK/NDK, OpenXR, Python runtime. BuildContext или fatal diagnostics.
1. Graph Analysis Загрузка .terminproj, parse scenes без side effects, resolution handles, include rules, module descriptors, duplicate UUID/name checks. build/graph.json, build/diagnostics.json.
2. Resource Compilation Compilation scenes, meshes, materials, shaders, pipelines, textures, fonts, audio, navmesh, voxels, foliage, Python modules. package/ с runtime artifacts.
3. Package Validation Schema version, relative paths, resource existence, shader matrix, duplicate UUIDs, material phases, scene refs, platform capabilities. Validated runtime package или structured errors.
4. Target Packaging Desktop bundle, Android APK, Quest/OpenXR APK. Runtime host получает уже проверенный package. Deployable artifact, logs, dependency report.

Target wrappers

Desktop, Android и Quest/OpenXR

Цель: один package contract, разные runtime hosts. Platform wrapper отвечает за доставку, lifecycle и native dependencies, а не за самостоятельную интерпретацию проекта.

Desktop

Bundle с termin_player, native libs и bundled Python при необходимости.

dist/<profile>/ app.json bin/termin_player lib/ package/ share/termin/
  • Только relative paths в app.json.
  • Dependency report: bundled libs vs system prerequisites.
  • Relocated bundle smoke как обязательный gate.

Android Surface

APK использует тот же runtime package и единый Android delivery contract.

dist/android/<profile>/ package/ apk/<app>-debug.apk logs/android-build.log
  • Native libs из Android SDK install.
  • Project-derived application id и label, если профиль не переопределяет.
  • Runtime host поднимает filesystem-like package root.

Quest/OpenXR

OpenXR lifecycle и swapchains поверх общего runtime package.

dist/quest_openxr/<profile>/ package/ apk/<app>-quest-openxr-debug.apk logs/quest-openxr-build.log
  • Требует openxr.enabled == true в SDK capabilities.
  • OpenXR app metadata берется из профиля.
  • Fail loudly, если сцена не XR-compatible.

Runtime contract

Runtime Package Contract

Runtime package target-independent там, где это возможно, и target-aware там, где нужны shader/runtime capabilities. Unknown major versions должны быть fatal. 

{
  "schema": "termin.runtime_package",
  "version": 2,
  "package_id": "com.example.game",
  "project_name": "Game",
  "entry_scene": "scene.json",
  "shader_artifact_root": "shaders",
  "targets": {
    "desktop": { "graphics": ["vulkan", "opengl"] },
    "android": { "graphics": ["vulkan"] },
    "quest_openxr": { "graphics": ["vulkan"], "xr": true }
  },
  "resources": [
    {
      "type": "shader",
      "uuid": "shader-uuid",
      "path": "shaders/shader-uuid.shader.json"
    }
  ],
  "diagnostics": []
}

Validated once, reused everywhere

  • Exporter tests.
  • Desktop build.
  • Android build.
  • Quest/OpenXR build.
  • Runtime loaders как defensive validation.
manifest.json *.shader.json *.tmat.json *.tmesh.json *.pipeline.json

Resource graph

Asset Graph и include policy

Статический scene graph не может надежно вывести dynamic resources: TcMaterial.from_name(...), gameplay-selected assets, DLC, plugins. Это должно быть явной частью профиля.

Default source categories

  • Entry scene, scene dependencies, prefabs.
  • Meshes, materials, textures, shaders, pipelines.
  • Audio, UI, fonts, navmesh, voxels, foliage.
  • Project module descriptors и stdlib runtime dependencies.
{
  "resource_includes": [
    {"type": "material", "glob": "Materials/**/*.material"},
    {"type": "texture", "glob": "Textures/UI/**/*.png"},
    {"type": "audio", "name": "MoveSound"}
  ]
}

dev

Broad include allowed, warnings for suspicious dynamic refs.

debug

Explicit dynamic includes required when not statically reachable.

release

No unresolved dynamic names, duplicate names are errors.

Observability

Diagnostics и SDK capability manifest

Логи нужны людям, diagnostics нужны tools/tests/UI. Capability manifest позволяет падать быстро до дорогой упаковки.

Structured diagnostic

{
  "level": "error",
  "code": "resource.missing",
  "path": "Scenes/Main.scene",
  "message": "Material uuid '...' is referenced but not included",
  "hint": "Add the material file or resource_includes entry"
}

SDK capabilities

{
  "version": 1,
  "platforms": {
    "desktop": {
      "vulkan": true,
      "opengl": true,
      "sdl": true,
      "python_runtime": true
    },
    "android": {
      "abis": ["arm64-v8a"],
      "vulkan": true,
      "python_runtime": false
    },
    "quest_openxr": {
      "abis": ["arm64-v8a"],
      "openxr_headers": true,
      "openxr_loader": true,
      "vulkan": true
    }
  }
}
Build возвращает non-zero на error и fatal. Любой fallback обязан иметь diagnostic code. Placeholder artifacts разрешены только в явно выбранной dev/smoke policy.

Interface

CLI и Editor integration

CLI становится canonical interface. Editor build UI остается тонким wrapper-ом: сохранить состояние, выбрать профиль, вызвать backend, показать logs/diagnostics/output paths.

Canonical commands

termin profiles [--project PROJECT]
termin profile PROFILE [--project PROJECT]
termin build PROFILE [--project PROJECT]
termin run PROFILE [--project PROJECT]
termin deploy PROFILE [--project PROJECT]
termin doctor PROFILE [--project PROJECT]

Editor should not

  • Traverse asset dependencies itself.
  • Call Gradle directly.
  • Mutate runtime package internals.
  • Depend on singleton preloaders for build correctness.

Quality gates

Tests and gates

Тесты должны покрывать не только happy path packaging, но и contract compatibility между exporter, Python loader, C++ runtime и platform wrappers.

Unit / contract

  • Profile schema validation.
  • Graph duplicate/missing diagnostics.
  • Runtime package schema validation.
  • Shader artifact target matrix.
  • Android/Quest routing with fake build scripts.

Integration

  • Export small scene package and load with Python loader.
  • Load same package with C++ termin-runtime.
  • Desktop bundle has no absolute source paths.
  • Relocated desktop bundle smoke.
  • Minimal Android and Quest APK assemble.

Device / manual

  • Android logcat smoke.
  • Quest/OpenXR runtime smoke.
  • Render pipeline visual sanity.
  • No shader/resource missing errors in logs.

Definition of done

Когда архитектура в порядке

Эти критерии полезны как чеклист для ревью миграции и как основание закрывать крупные технические задачи.

Success criteria

  • Каждый termin build PROFILE идет через один project build backend.
  • Build output объясним из profile + source tree + SDK capabilities.
  • Runtime package schema-validated и loader-compatible.
  • Desktop/Android/OpenXR разделяют один package contract.
  • Editor build actions являются thin wrappers.
  • Missing resources падают structured diagnostics, а не device-only сюрпризами.

Open questions

  • Runtime scene format: editor JSON или compiled scene после стабилизации handles?
  • Когда переводить mesh/material artifacts из JSON transition format в binary sidecars?
  • Python on Android: non-goal сейчас или будущий optional target?
  • Plugin/resource compilers: SDK entry points, project plugins или оба источника?
  • Release dependency policy: vendor libs или host prerequisites?
