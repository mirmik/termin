# Multilanguage Component/Pass/Widget Lifetime Model

## Категории мультиязычных классов tc_component, tc_pass, tc_widget

Документ описывает модель контроля лайфтайма общей категории классов.

Каждая из этих трёх категорий, несмотря на различие доменных областей следует единой модели. Есть владеющий тимп-коллекция:
tc_entity, tc_pipeline, tc_ui_document, вне которых классы tc_component, tc_pass, tc_widget не имеют какого либо смысла. Никаких причин перемещать классы между владеющими контейнерами также нет.

При этом tc_component, tc_pass, tc_widget имеют наследников в разных языках. Реализованы c++/python, возможно добавление иных языков. Реализация наследования сделана через композицию. Добавление базовой структуры в тело класса-наследника и таблицы виртуальных методов. 

Также область применения всех трёх категорий предлполагает возможность и необъходимость сериализации. Предполагается также возможность создания собственных типов наследников в пользовательском коде и возможность горячей перезагрузки.

Всё это указывает на то, что типы tc_component, tc_pass, tc_widget должны стремиться к единому паттерну построения. Далее в тексте, когда речь идёт об одном из трёх типов, в большей степени предполагаются все три. 

## О лайфтайме и удалении. 

Исторически лайфтайм tc_component строился на счётчике ссылок. Модель счётчика ссылок выглядит избыточной, поскольку единственным владельцем является "владеющий-контейнер". Кроме того модель счётчика ссылок вносит изрядную путанницу в порядок, кто и когда должен освобождать ссылки и что делать с освобождённым объектом.
Вместо этого предполагается модель с единственным deleter без счётчика ссылок. Общее предложение такого, что код, создающий компонент должен указать в deleter способ унитожения этого компонента, который контейнер будет при необъходимости вызывать. 

На текущий момент я вижу три модели передачи контроля над компонентом:
Пайтон код создаёт python-компонент или обёрнутый c++-компонент. При передаче управления он делает IncRef, и передаёт deleter с DecRef.
C++ код создаёт c++ компонент через new и передаёт c честным delete.
С++ код статически (в теле объекта) инстанцирует компонент, не передавая управление его лайфтаймом под контроль контейнера. Такой объект подключается отдельной операцией borrowed attachment; каноническая операция adopt всегда означает передачу владения и всегда требует непустой deleter.

`tc_widget` закрепляет этот контракт на уровне API: `tc_widget_init_unowned` только
инициализирует объект и не назначает владельца, `tc_ui_document_adopt_widget`
атомарно принимает объект вместе с deleter, а
`tc_ui_document_attach_borrowed_widget` является отдельным явно невладеющим
путём. Lifecycle callback `on_destroy` вызывается до инвалидации объекта, но
освобождение памяти выполняется только deleter после удаления всех ссылок из
контейнера и увеличения generation handle.

`tc_pass` следует той же owned-модели без borrowed attachment. Создатель
инициализирует pass через `tc_pass_init_unowned` и задаёт способ удаления;
`tc_pipeline_adopt_pass` атомарно переносит pass и deleter в один pipeline slot.
Удаление, замена и уничтожение pipeline сначала убирают slot из наблюдаемого
состояния, затем вызывают lifecycle `tc_pass_vtable::destroy` и ровно один раз
deleter. Перестановка внутри pipeline выполняется отдельной операцией move и не
разрушает pass. Ссылки, возвращаемые lookup API, являются borrowed и не должны
переживать мутацию pipeline. Старого протокола `retain`/`release`/`drop` у
`tc_pass` больше нет.

Для Python pipeline хранит один `Py_INCREF`, а его deleter выполняет один
`Py_DECREF`; unattached pass не удерживает сам себя. Для C++ deleter выполняет
`delete` конкретного объекта, для C# освобождает переданный `GCHandle`.
Неуспешная передача владения откатывается через тот же deleter. Таким образом,
языковой lifetime-hold принадлежит контейнеру, а не borrowed-обёрткам.

## О фабриках и inspect. 

Полезна возможность создания класса, исходжный код которого расписан в другом языке. Для этого tc_component имеет фабрики, вынесенные в рантаймы языков, а TcEntity методы add_component, get_component, работающие через нейтральный TcComponent. Предполагается, что подобную практику стоит распространить на tc_pass, tc_widget, а то и вовсе сделать фабрики общей инфраструктурной единицей. Тоже касается и поддержке всеми тремя категориями иинспект-системы.

## Общее направление.

Цель инициативы в приведении трёх категорий к общей мультиязычной модели с ядром в си коде. Технически это должно в дальнейшем облегчить интеграцию движка в другие языки программирования и построении систем, разные части которых написаны в разных языках.
