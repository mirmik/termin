## #47 qopt effective dynamic system

Status: implemented, ready for test.

Finding:

- `MatrixAssembler.solve_Adxx_Cdx_Kx_b()` called a non-existent
  `assemble_Adxx_Cdx_Kx_b()`.
- The method attempted to add `C @ x_dot` and `K @ x` vectors to `Ad`, then
  ignored that effective system and called a non-existent generic `solve()`.

Fix:

- The method now assembles through `assemble_dynamic_problem()`.
- It solves `Ad * x_ddot = b - C * x_dot - K * x` using `_solve_system()`.
- It validates `x_dot` and `x` against the system DOF shape and writes the
  solved acceleration back to variables through `set_solution_to_variables()`.
- Added a focused dynamic test with non-zero mass, damping, stiffness and load
  terms. The test verifies that changing `x_dot`/`x` changes the acceleration.

Verification:

- `.venv/bin/python -m pytest termin-qopt/tests/fem`
- Result: `80 passed`.

## #81/#100 nanobind leak reports

Status: implemented for the reproduced Linux lightweight-import leaks; #100's
original Windows-only public `tcbase.Vec3` symptom was stale in the current tree

  metadata in native registries. Those registries held Python/nanobind objects
  until interpreter finalization, producing leak reports for `TcRenderState`,
  `TcMaterial`, `TcMesh`, `ComponentRegistry`, render pass types and functions.
- Manual cleanup of `ComponentRegistry` plus the corresponding
  `InspectRegistry` type entries removed the report, confirming the ownership

  `termin_voxel_components.voxelize_enums`.
- Updated `termin.voxels` and `termin_voxel_components` lazy facades so enum
  access no longer imports the full drawable component module.
- Added `PythonComponent` atexit cleanup that unregisters only Python-backed
  component types and inspect types registered through `PythonComponent`.
- Added subprocess smoke tests asserting no `nanobind: leaked` output for
  `from termin.voxels import VoxelizeSource` and
  `from termin.navmesh.builder_component import NavMeshBuilderComponent`.

## #77/#34/#119 player shutdown/finalization

Status: implemented, ready for test.

Findings:

- Source player did not run shutdown after `initialize()` failed before scene
  start. The Vulkan/offscreen window failure path left process-global render
  pipelines alive until Python finalization.
- Python-authored render passes created through the native pass factory kept an
  extra factory-owned Python reference; destroying the pipeline released only
  the pipeline retain.
- SDK source-player still aborted after the pass leak was fixed because the
  process-global GLSL preprocessor held a Python fallback callback until the
  `GlslPreprocessor` static destructor ran after `Py_Finalize`.
- C++ `termin_player` let Python handle SIGINT in the embedded interpreter,
  which could raise inside render callbacks instead of requesting a host quit.
- The old termin-chess shutdown report is now mostly stale: the C++ host path
  loads, unloads, and destroys the scene/core cleanly with the current SDK.

Fix:

- `PlayerRuntime.run()` now calls `shutdown()` when initialization fails.
- `PlayerRuntime.shutdown()` tolerates partial initialization, destroys live
  render pipelines, and releases shader runtime callbacks before Python
  finalization.
- `tc_pass_bindings` now releases the factory-owned Python pass reference when
  an unowned factory-created pass is destroyed.
- Python frame pass registrations can now be explicitly unregistered; the
  Python facade unregisters Python pass types at exit.
- Materials binding now exposes `unregister_glsl_preprocessor()`, and
  `termin.shader_runtime` uses it from explicit shutdown and `atexit`.
- C++ player host installs SIGINT/SIGTERM handlers after Python initialization;
  handlers only set an atomic flag, and the main loop converts it to
  `request_quit(128 + signal)`.

Verification:

## #108 Windows editable install locked native modules

Status: implemented, ready for test.

Finding:

- Editable installs are intentionally sequential, so a Windows lock failure in
  one package stops later packages from being synced. Before this pass the
  failure surfaced as a raw pip/WinError message without saying which package
  stopped the SDK sync or which processes might hold loaded native modules.

Fix:

- `install_pip_packages()` now prints a failure summary with the failed package
  index and explains that remaining packages were not installed.
- On Windows editable installs, it scans the failed package for `.pyd`/`.dll`
  artifacts, queries `tasklist /m <module>` for loaded-module owners, lists
  Python/Termin processes that may hold native modules, and prints a concrete
  rerun command.
- Added a mocked Windows unit test for this diagnostic path.

Verification:

- `.venv/bin/python -m pytest termin-build-tools/tests/test_sdk_orchestrator.py`
- `./run-tests.sh`

## #48 fast-math and infinity semantics

Status: implemented, ready for test.

Policy:

- Repository-owned engine/runtime targets must not enable global fast-math
  flags because collision, lighting, plotting, ray queries, and debug paths may
  intentionally observe IEEE NaN/Inf sentinel values.
- Fast-math may be scoped to a leaf numeric kernel only when that kernel's
  contract explicitly excludes NaN/Inf inputs and outputs and has tests for
  that contract.

Fix:

- Removed global `-ffast-math` from `termin-app/cpp/CMakeLists.txt` release
  target options.
- Replaced MSVC `/fp:fast` with `/fp:precise` for the same runtime targets.
- Re-enabled clang's `nan-infinity-disabled` diagnostic in `run-lint-cpp.sh`.
- Updated `docs/linting.md` with the fast-math policy.
- Added a C++ sentinel test covering empty `UnionCollider` infinity distance
  and directional light infinity distance.

Verification:

- `./run-tests.sh`
- `rg -n "ffast-math|/fp:fast|finite-math" build/Release-tests/compile_commands.json build/Release/compile_commands.json`
