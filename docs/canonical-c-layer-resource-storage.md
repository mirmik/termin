# Ресурсный слой ядра termin.

## Какую проблему решает

Движок termin предполагает работу в мультиязычной среде и должен быть готов к постоянным переходам между рантаймами, через слой интеропа. 
Обычно движение через интероп слои ведёт к сложным политикам управления временем жизи объектов и запутанным схемам владения.

## В чём идея

Вместо того, чтобы решать проблемы владения на границах, движок termin вводит каноничный слой хранения ресурсов на стороне си ядра. Доступ к ресурсам осуществляется по слабым generation-handle ссылкам, с явным контролем лайфтайма. Шаблонная схема такова, что в ядре создаётся глобальный пул объектов определённого типа, с ключём вида (int handle, int generation). В зависимости от требований к ресурсу эта схема может существенно варьироваться, но неизменным остаётся то, что storage объектов принадлежит глобальному слою на стороне си ядра, а доступ осуществляется через хэндлы на основе индексов, не требующих переноса процедурного владения через интероп слой и устойчивых к изменению размеров пула. Для уничтожения объектов предпочтителен явный destroy.

Глобальные пулы инициализируются при старте процесса и уничтожаются (если это возможно) при завершении. Пулы считаются process-owned. Логическим лайвтаймом пулов полагается время жизни процесса. Все прочие системы движка включая EngineCore должны полагать глобальные пулы существующими всегда и не пытаться управлять их инициализацией.

Об этом ресурсном слое следует думать как об общей ресурсной памяти движка. Системы движка шарят между собой ресурсы, складывая их в ячейки и передавая между собой хэндлы на эти ресурсы.

Необходимость явного контроля лайфтайма является недостатком схемы, однако позволяет сделать контроль времени жизни объекта прозрачным.
При работе с ресурсным слоем требуется проверять актуальность хэндлов через сверку актуального generation. Обнаружение протухшего хэндла должно явно логироваться и отслеживаться.
Общий ресурсный слой также даёт возможность контроля за утечками ресурсов через систему мониторинга Core Registry.

tc_shader

## О виртуальных наследниках типов, время жизни которых управляется си слоем.

Существует ещё один паттерн управления, повторяемый в движке. Он относится к ситуации, когда хранимый в глобальном пуле ресурс служит/управляет абстрактной коллекцией. Рассмотрим этот паттерн на примере пары tc_entity - tc_component. Entity владеет списком components не имеющих никакого смысла без этого самого entity. При этом tc_component является базовым си типом. Конкретные типы-компоненты "паследуются" от него, беря tc_component в свою композицию и размечая vtable для того, чтобы entity имел возможность взаимодействия. Т.к. время жизни компонента не может превышать время жизни entity, но entity не ведает способа создания, а следовательно и правильного способа освобождения компонента - каждому компоненту на этапе подключения в коллекцию передаётся актуальный deleter. 

Такая схема позволяет гибко выбирать способ создания и сохранения компонентов не осложняя конструкцию сложными схемами владения. Достаточно лишь гарантировать, что условная "фабрика" удалит компонент по требованию entity. 

Схеме следуют (должны следовать. в скобках указан владелец):

tc_component (tc_entity)
tc_pass (tc_pipeline, рантайм объект)
tc_widget (tc_ui_document)
tc_render_surface (tc_display)

`tc_ui_document` хранится в process-owned registry и пересекает C/C++/Python
границы только как `tc_ui_document_handle`. Вложенный `tc_widget` хранит handle
владельца, а не адрес документа; уничтожение документа сначала освобождает все
виджеты их зарегистрированными deleter-ами и лишь затем инвалидирует document
handle. Для намеренных проверок используется тихий `is_valid`/`is_alive`, а
операционные resolve-paths логируют протухшие generation handles.

Для pipeline две роли образуют один паттерн, но не один storage type:
`tc_pipeline_template` — каноничный неизменяемый/versioned шаблон, а
`tc_pipeline` — его mutable execution instance, владеющий живыми `tc_pass` и
их deleter'ами. Scene и asset удерживают strong handle шаблона; render topology
создаёт отдельный instance для конкретного runtime attachment.
Созданный напрямую runtime-only `tc_pipeline` может не иметь шаблона вообще;
такой instance обязан возвращать invalid template handle, а не создавать
пустой анонимный canonical template как побочный эффект.
